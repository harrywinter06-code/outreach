# Swarm Z2.5 — Close the Revenue Loop Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the gap left by Z2. Today, businesses are spawned, tracked, and selected — but no agent works on them, no Stripe payment routes to their revenue ledger, and no skill call knows which business it belongs to. After Z2.5 the loop is genuinely closed: a business is spawned with a genome → an LLM cycle uses that genome to author + publish + monetise → real customer pays via business-attributed Stripe link → webhook posts revenue to `business_revenue` → `compute_business_fitness` rises → genome graduates to the template pool → future spawns inherit the winning shape. The substrate becomes self-improving on the actual £-signal.

**Architecture:** Three layers, in dependency order. Layer A (attribution) is foundation — adds `business_id` to `skill_calls`, threads it through `SkillCtx` and the directive router, and surfaces per-business activity score in the fitness formula. Pure plumbing, no behaviour change yet. Layer B (execution) introduces a `business_cycle_runner` scheduler loop that wakes one business per tick, builds a genome-scoped prompt with role-filtered skills, gets an LLM action, and dispatches it through the existing directive router with `business_id` attribution. Each cycle must produce one concrete artifact (post URL, email captured, payment link created); cycles producing nothing N times in a row become a cull signal. Layer C (revenue capture) extends the existing FastAPI dashboard with a Stripe webhook endpoint; signature-verifies; reads `metadata.business_id` from the charge; routes to `BusinessStore.record_revenue` with `is_self_paid=False`. The `stripe_create_payment_link` skill is modified to require and embed `business_id` in its metadata, so the webhook always has the attribution.

**Tech Stack:** Python 3.13 (existing), `asyncpg` (existing), `stripe` SDK ≥7.0 (already in deps for `_LivePayments`), `fastapi` (already in dashboard), `pytest` (existing). No new pip dependencies. New tables added to existing `db.py:init_schema` path — idempotent `IF NOT EXISTS` everywhere.

**What this is NOT:**
- Not a per-business agent process (each business stays a row, not a SOUL.md file). Z3+ if proven valuable.
- Not domain registration / asset acquisition (`business_assets` table stays decorative). Z3.
- Not a capital allocator that rebalances budgets across businesses. Z3.
- Not a genome mutation operator (Z4) — sampling stays "70% template, 30% seed" with no crossover yet.

**Pre-mortem:** The single most likely failure is that `business_cycle_runner` cycles produce LLM-fluent JSON but never converge on a concrete artifact — agents narrate planning forever instead of publishing, capturing, or charging. Mitigation is the **artifact gate**: every cycle's parsed action must produce a measurable side-effect (a URL written to `businesses.metadata.last_artifact_url`, an email row in `business_leads`, a stripe payment link id in `businesses.metadata.payment_link_url`); cycles that produce nothing N=3 times in a row penalise fitness and shorten the kill clock. Second most likely: the Stripe webhook endpoint fails signature verification because the operator misconfigures the webhook secret, so live payments arrive but never get recorded — mitigated by structured-log every rejected webhook with the signature error so the operator can debug from `docker compose logs`. Third: business_id leaks into a global skill call (e.g. executive cycle calls a skill while last business cycle's ctx was still cached) — mitigated by making `business_id` a per-call kwarg explicitly passed to `make_live_ctx`, never read from a shared mutable.

### Task C2: Webhook server

- [ ] **Step 1: `stripe_webhook.py`**

```python
from fastapi import APIRouter, Request, HTTPException
import stripe

router = APIRouter()

@router.post("/webhook/stripe")
async def stripe_webhook(request: Request):
    payload = await request.body()
    sig = request.headers.get("stripe-signature", "")
    try:
        event = stripe.Webhook.construct_event(
            payload, sig, settings.stripe_webhook_secret,
        )
    except (ValueError, stripe.error.SignatureVerificationError) as exc:
        logger.warning("Stripe webhook signature failed: %s", exc)
        raise HTTPException(status_code=400, detail="signature_invalid")

    if event["type"] in ("charge.succeeded", "payment_intent.succeeded"):
        obj = event["data"]["object"]
        biz_id = (obj.get("metadata") or {}).get("business_id")
        if not biz_id:
            logger.warning("Stripe %s has no business_id metadata: %s", event["type"], obj.get("id"))
            return {"ok": True, "routed": False}
        amount_gbp = float(obj["amount"]) / 100.0  # Stripe uses minor units
        # Currency check — only route GBP for now
        if obj.get("currency", "").lower() != "gbp":
            return {"ok": True, "routed": False, "reason": "non_gbp"}
        await BUSINESS_STORE.record_revenue(
            business_id=biz_id, amount_gbp=amount_gbp,
            source="stripe", external_id=obj["id"],
            is_self_paid=False,
            metadata={"stripe_event_id": event["id"], "event_type": event["type"]},
        )
    elif event["type"] == "charge.refunded":
        obj = event["data"]["object"]
        biz_id = (obj.get("metadata") or {}).get("business_id")
        if biz_id:
            await BUSINESS_STORE.record_revenue(
                business_id=biz_id,
                amount_gbp=float(obj["amount_refunded"]) / 100.0,
                source="stripe", external_id=f"refund_{obj['id']}",
                is_refund=True,
            )
    return {"ok": True}
```

## Pre-mortem revisited (per task)

- **Task A:** Most likely failure is the `_handle_skill_call` signature change breaking some legacy caller. Mitigation: business_id is kwarg with default None, signature is additive. Run full suite after the change.
- **Task B:** Most likely failure is the LLM returns plausible JSON that doesn't match any registered skill, the cycle dispatches "nothing," stall counter never resets even for busy-looking cycles. Mitigation: in the cycle runner, after dispatching, wait briefly for the skill's result in the inbox; only count stall=0 if the result has `ok=True`. Don't count action-issued as artifact — count action-completed-successfully as artifact.
- **Task C:** Most likely failure is the webhook URL is unreachable from Stripe because Caddy isn't routing `/webhook/stripe` through to the dashboard FastAPI app. Mitigation: in Caddyfile, ensure the existing reverse-proxy block forwards all paths, not just `/`. Verify with curl from a non-VPS machine before flipping live mode.
